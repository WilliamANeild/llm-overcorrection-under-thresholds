When LLMs Revise Past the Point of Diminishing Returns

Liam Neild

Core questions
  • Do LLMs revise because the output genuinely needs it, or because of conversational pressure?
  • At what point does revision stop improving quality?
  • What does unnecessary revision cost, and can it be prevented?
The Curve
  • Where does output quality plateau across turns?
  • Does the plateau differ by task domain?
  • Do models keep revising past diminishing returns?
The Mechanism
  • Is revision driven by compliance or quality judgment?
  • Does one fresh attempt beat five rounds of iteration?
  • Does the model prefer its own first draft in a blind test?
The Cost
  • How many tokens go to revisions that add no quality?
  • Do responses bloat and lose diversity over rounds?
  • Do different models converge toward similar outputs?
  • Do later turns violate original task constraints?
The Fix
  • Does specific critique outperform generic "improve this"?
  • Can the model spot its own overcorrection when asked?

Step 2 — Blind same-model evaluation

Instance
A separate instance of the same model evaluates each turn's output
Each turn's output is extracted from the working conversation and shown to a fresh instance of the same model with zero conversation history. The evaluator is told "someone else produced this for the following task" and sees only the original task description and the output. It has no knowledge of the revision history or how many turns have occurred.
The evaluator is asked: "Does this output fulfill the task objective well enough that further revision would be diminishing returns relative to the token cost?"

Step 3 — Measuring the overcorrection gap
At each turn, we compare two signals. The working model inside the conversation decides whether to revise. The blind evaluator, seeing the same output with fresh eyes, independently judges whether the output is already sufficient. When the blind evaluator says sufficient but the working model keeps revising, the difference is overcorrection. The quality gained per token spent across turns is the Revision Yield.
Supporting tests
One-Shot Ceiling
The model is asked to produce the best possible output in a single attempt, with no revision history. If this one-shot output matches or beats five rounds of iterative revision, then the entire revision loop added nothing of value.
Reversibility
The model is shown its turn 1 output and turn 5 output side by side in a blinded A/B comparison, with no labels indicating which came first. If it prefers its own first draft, the revisions actively made the output worse.
Targeted Feedback
When the blind evaluator genuinely flags an issue, we test whether specific, actionable critique produces better revisions than a generic "please improve this" prompt. Measures whether feedback quality matters.
Self-Reflection
After five turns of revision, we ask the model in the same context: "Which version would you recommend to the user?" Tests whether the model can recognize when its own revisions went too far.
